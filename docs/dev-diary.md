## F1 — Backend Skeleton + Postgres async + Health endpoint

**Date**: 2026-05-12
**Branch**: `refactor-clean`
**Duration**: 1 session

### Scope

First working iteration of the FastAPI backend. Goal: a cleanly structured project that
starts, connects to Postgres, and exposes a `/health` endpoint. No domain models yet.

### Architecture decisions locked in this phase

| Decision | Choice | Rationale |
|---|---|---|
| Python package manager | **uv** workspace (root + backend member) | 10–100× faster than pip; single `uv.lock` for entire monorepo |
| Postgres driver | **asyncpg** + SQLAlchemy 2.0 async | Non-blocking I/O essential for long-running forecast jobs |
| Migrations | **Alembic** (async mode, no psycopg2 needed) | Schema reproducibility requirement |
| JWT library | **PyJWT** (replaces python-jose) | jose had unpatched CVEs; PyJWT actively maintained |
| Logging | **structlog** JSON output | Machine-readable audit trail for the reproducibility requirement |
| Dev workflow | **uvicorn --reload** + bind-mount | Fast iteration — code changes reload without image rebuild |
| RBAC roles | `admin` / `researcher` / `viewer` (from F2) | Aligns with TFG security requirements |
| MongoDB driver | **motor** async | Non-blocking; already in root pyproject |
| Config management | **pydantic-settings** (`.env` → `Settings` class) | Single source of truth, validated at startup |

## F2 — Authentication (JWT + RBAC + admin seed)

**Date**: 2026-05-12
**Branch**: `refactor-clean`
**Duration**: 1 session

### Scope

Full authentication layer on top of F1. Goal: users can sign up, log in, receive a JWT,
and hit a protected endpoint. RBAC roles (`admin` / `researcher` / `viewer`) are enforced
from the first migration. Admin account is auto-seeded at startup.

### Architecture decisions locked in this phase

| Decision | Choice | Rationale |
|---|---|---|
| Password hashing | **bcrypt** via the `bcrypt` package | Industry standard; no external C deps risk |
| JWT signing | **PyJWT** HS256, 7-day expiry | Short enough to be practical; configurable via `JWT_EXPIRES_MIN` env var |
| Role injection | Blocked at signup level — role is ALWAYS `viewer` for public signup | Prevents privilege escalation; admin must be seeded or promoted by another admin |
| Postgres enum type | `sa.Enum("admin","researcher","viewer", name="user_role")` in `create_table` | Cleaner than manual `CREATE TYPE` + `create_type=False`; Alembic handles creation and drop in single transaction |
| Admin seed | Idempotent in `lifespan` — checks for existing email first | Safe to restart container without duplicating the admin row |
| Test event loop | `asyncio_default_test_loop_scope = "session"` + session-scoped `client` | SQLAlchemy async engine is a module-level singleton; per-test loops cause "Future attached to different loop" errors |
| Test isolation | `TRUNCATE TABLE users RESTART IDENTITY CASCADE` before each test | Guarantees each test starts with a clean state; test-suite is re-runnable without resetting Docker volumes |

### Issues encountered

| Issue | Root cause | Solution |
|---|---|---|
| `type "user_role" already exists` | Migration had both `op.execute("CREATE TYPE...")` AND `sa.Enum(..., create_type=False)`, where SQLAlchemy's `_on_table_create` event fired a second `CREATE TYPE` despite `create_type=False` | Remove manual `op.execute`; use `sa.Enum(..., name="user_role")` with default `create_type=True` — single path |
| `alembic/` package shadowing | `PYTHONPATH` included `backend/`, which contains `alembic/` dir → Python found our directory instead of the installed package | Rename `alembic/` → `migrations/`; update `alembic.ini` to `script_location = %(here)s/migrations` |
| `%(here)s` in alembic.ini broke when run from `/app` | alembic couldn't find `script_location` if ini path was relative | Use `-c /app/tfg-arquitectura/backend/alembic.ini` + `%(here)s` resolves relative to the ini file |
| `docker compose run -w /app/...` on Windows Git Bash | Git Bash converts `/app` → `C:/Program Files/Git/app` | Run via PowerShell with `//app/...` double-slash, or use WSL |
| `email-validator` not installed | Missing dep; Pydantic `EmailStr` requires it at import time | Added `email-validator>=2.1` to `Dockerfile` |
| `pytest` not in image | Dockerfile only installed runtime deps | Added `pytest>=8.0` + `pytest-asyncio>=0.23` to Dockerfile |
| "Future attached to a different loop" in tests | Per-test event loops (pytest-asyncio default) incompatible with module-level SQLAlchemy async engine | `asyncio_default_test_loop_scope = "session"` in pytest config; session-scoped `client` fixture |
| First test fails with 409 even on clean run | `clean_db` fixture was teardown-only (yield first) so first test saw previous session's data | Moved TRUNCATE before `yield` — cleanup runs before each test |

### What went well

- `require_role(*roles)` factory pattern (from Filmly) works cleanly with FastAPI's DI; adding a new role check is one line
- Admin auto-seed with idempotency check is robust — safe to restart the backend repeatedly
- PyJWT + bcrypt have zero C build steps in the Docker image (bcrypt ships pre-compiled wheels)
- 11/11 integration tests green on first clean run

## F3 — Domain models, ETL loader, read-only catalog API

**Date**: 2026-05-12
**Branch**: `refactor-clean`
**Duration**: 1 session

### Scope

Define the full domain schema (7 tables beyond `users`), run the second Alembic migration,
load the research parquets into Postgres, and expose read-only endpoints that the frontend
will consume to build the experiment UI.

### Architecture decisions locked in this phase

| Decision | Choice | Rationale |
|---|---|---|
| Observation storage | One row per `(series_id, timestamp)` with a unique index | Enables conflict-free upserts; simple to query ranges |
| ETL concurrency | Sequential per-series, single session | Keeps it simple; 6 k rows load in <2 s — no batching needed yet |
| Bulk insert | Inline VALUES string (no `executemany`) | asyncpg prepared-statement overhead makes `executemany` slower for large homogeneous inserts |
| Model catalog | Seeded by ETL script, not in migration | Catalog can be updated without schema changes |
| JSONB config on Experiment | Yes — stores hyperparameter dict | Avoids wide table for model-specific params; queried rarely |
| Observations endpoint | Paginated (`limit`/`offset`) | Series can have 300+ points; avoid dumping all in one response |
| `::type_cast` in raw SQL with asyncpg | AVOID — `:x` before `::type` confuses SQLAlchemy's param parser | Use ORM for enum inserts; raw SQL only for non-enum values |

**Date**: 2026-05-12

**Duration**: 1 session

### Scope

Full CRUD lifecycle for experiments. Users with `researcher` or `admin` role can create and
delete experiments; any authenticated user can list and read their own; admins see everything.

### Architecture decisions locked in this phase

| Decision | Choice | Rationale |
|---|---|---|
| Role guard on POST/DELETE | `require_role(researcher, admin)` | Viewers are read-only consumers; only researchers run experiments |
| GET list scope | Non-admin sees own only; admin sees all | Least-privilege default; admin oversight is an explicit requirement |
| GET detail ownership | `user_id == current_user.id OR admin` | 403 (not 404) on access denial — reveals that the resource exists, but this is acceptable since experiment IDs are sequential and not secret |
| `selectinload(Experiment.runs)` | Eager-load runs in detail endpoint | Single query vs N+1; runs list is small (rarely >50) |
| `ExperimentDetailOut` extends `ExperimentOut` | Pydantic model inheritance | Detail adds `runs: list[RunOut]`; list endpoint stays lean |
| Horizon validation | `ge=1, le=60` on `ExperimentCreate.horizon` | Mirrors the max horizon used in tfg-forecasting research (12–24 months); 60 is a safe upper bound |
| Test fixtures for roles | ORM inserts in `tests/integration/conftest.py` | `signup` only creates viewers; direct DB insert is the simplest way to seed arbitrary roles |

## F5 — Forecast runs: adapter pattern + synchronous execution

**Date**: 2026-05-12
**Branch**: `refactor-clean`
**Duration**: 1 session

### Scope

First working forecast: a user triggers a run on an experiment, the platform trains a model
against held-out test data, persists predictions + metrics, and returns everything in one
response. F5 covers the naïve seasonal baseline and SARIMA — both match the research
baselines exactly. Foundation models (TimesFM, Chronos-2, TimeGPT) are deferred to F7.

### Architecture decisions locked in this phase

| Decision | Choice | Rationale |
|---|---|---|
| Adapter pattern | `ForecastAdapter` Protocol in `app/forecasting/` | tfg-forecasting scripts are standalone research code; we replicate the model logic in clean callable adapters — zero changes to the research repo |
| Train/test split | Hold out last `horizon` observations as test set | Mirrors the rolling-window backtesting from the research; user sees "how well would this model have done on recent data" |
| SARIMA orders | `(0,1,1)(0,1,1)12` from auto_arima on Spain IPC | Reuses the validated orders from `02_sarima_seasonal.py`; no re-selection overhead |
| Synchronous execution | `run_in_executor(None, adapter.run, inp)` | SARIMA fit is CPU-bound; running in thread pool avoids blocking the async event loop. F6 will move to BackgroundTasks for long-running models |
| Metrics stored | MAE, RMSE, MAPE | MAE/RMSE match `shared/metrics.py`; MAPE is added for intuitive percentage interpretation |
| statsmodels freq fix | Pass `y_train.values` (numpy) instead of dated Series | statsmodels 0.14+ fails with `freq="MS"` when converting DatetimeIndex to PeriodIndex internally (`"for Period, please use 'M' instead of 'MS'"`) |
| Run lifecycle (F5) | `pending → running → done/failed` all in one request | Sufficient for sync models; F6 decouples trigger from execution via BackgroundTasks |

## F6 — Role promotion, background runs, runs list, MCP stub

**Date**: 2026-05-12
**Branch**: `refactor-clean`
**Duration**: 1 session

### Scope

Move forecast execution off the request thread (FastAPI `BackgroundTasks`), give admins a way
to promote users, expose a per-experiment runs history, and stub the `use_mcp` flag so it does
not break execution before F8 wires the real client.

### Architecture decisions locked in this phase

| Decision | Choice | Rationale |
|---|---|---|
| Background execution | FastAPI `BackgroundTasks` (in-process) | Simpler than Celery/RQ for a TFG-scale workload; sufficient until concurrent long runs (Chronos-2) become a bottleneck in F7+ |
| Background task DB session | Opens its own `AsyncSessionLocal()` | The request's session is closed once the response is sent; sharing it would error mid-task |
| Response code on trigger | `202 Accepted` (not 201) | Semantic match: resource accepted, not yet completed |
| Trigger response body | `RunOut` (no predictions/metrics) | Body is serialised before the background task runs; predictions/metrics aren't there yet — clients GET them after polling |
| `use_mcp` handling | Log-only stub: `log.info("mcp_requested_stub", ...)` | Doesn't break F5 flow; F8 will replace the stub with the actual MCP client call |
| Role demotion safeguard | Admin cannot demote their own account | Prevents lockout if the seed admin is the only one |
| Token role vs DB role | DB role is authoritative — `get_current_user` re-fetches user from DB | Promoting a user takes effect immediately; existing tokens see the new role on next request |

## F7 — Foundation model adapters (TimesFM, Chronos-2, TimeGPT)

### Scope

Implements the three foundation-model adapters, completing the adapter registry for all model

### Architecture decisions locked in this phase

| Decision | Choice | Rationale |
|---|---|---|
| Optional imports (try/except) | Each adapter catches `ImportError` at call time | Keeps the registry lightweight at startup; models that aren't installed don't break boot |
| Lazy model loading (thread-safe) | Module-level `_MODEL + _MODEL_LOCK` (double-checked locking) | TimesFM and Chronos-2 are 200M-param models; loading once and reusing avoids repeated 5-second startup costs per run |
| p50 from Chronos-2 quantiles | `quantiles[0, _P50_IDX=10, :]` | Chronos-2 returns 21 quantiles (0.01–0.99); index 10 = median (p50), matching the `y_pred` convention used in research scripts `03_chronos2_C0.py` |
| TimeGPT fail-fast on missing key | Check `os.getenv("NIXTLA_API_KEY")` before constructing `NixtlaClient` | Clear error message rather than opaque HTTP 401 from Nixtla's servers |
| Unsupported model slug renamed | `"timegpt"` → `"not-a-model"` in `run_catalog` fixture | `timegpt` is now registered; the "not implemented" test needs a slug that's genuinely absent from the registry |
| Train/test split | Same as F5 — hold out last `horizon` obs as test | Consistent evaluation protocol across all adapters |

## F9b — Ridge-exog adapter (backend completion)

**Date**: 2026-05-14
**Branch**: `refactor-clean`
**Duration**: 1 session (continuation of F9)

### Scope

Closes the last gap in the adapter registry: `ridge-exog` was seeded in the model catalog
(F3) but never registered. Implements Ridge regression with ECB exogenous features using
a direct multi-step forecasting strategy. Falls back to AR-only mode when the
`features-exog` dataset is not loaded (test environment / missing ETL).

### Architecture decisions

| Decision | Choice | Rationale |
|---|---|---|
| Direct multi-step | One Ridge model per forecast step d=1…H | Avoids error accumulation from recursive forecasting; matches the C1_inst rolling-window evaluation in the research |
| AR lags | [1, 2, 3, 12] | Captures recent trend (lag 1-3) + seasonal cycle (lag 12) |
| Exog features | `dfr, mrr, dfr_diff, dfr_lag3, dfr_lag6, dfr_lag12` | All are pre-computed from ECB rates; no look-ahead bias since ECB rate decisions are public same-day |
| StandardScaler per model | Fit on training window, transform forecast features | Prevents scale sensitivity in Ridge penalty; required by methodology notes |
| Graceful AR-only fallback | `exog=None` → `live_exog_cols=[]` → pure AR features | Test containers don't load the real parquet data; adapter still produces valid forecasts |
| `ForecastInput.exog` field | Added `exog: pd.DataFrame | None = None` to dataclass | Backward-compatible extension — all existing adapters ignore the new field |
| Exog loading location | `_load_exog_for_ridge()` in `runs.py`, called from `_execute_forecast` | Keeps adapters as pure synchronous functions; DB access stays in the async layer |
| `scikit-learn>=1.4` | Added to Dockerfile | Required for `Ridge` + `StandardScaler`; was missing from deps |

| Routing | React Router v6 with `<ProtectedRoute>` HOC | Standard; `<Outlet>` keeps shared layout |
| Charts | Recharts (not Plotly) | 60 kB vs 1 MB; native React components work seamlessly with shadcn theme variables |
| Component install path | shadcn MCP via `.mcp.json` at frontend root | Per-project MCP isolated from global config |
